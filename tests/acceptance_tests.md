## АС-001: Регистрация и авторизация пользователя

**Цель:** Система обеспечивает безопасную регистрацию и вход.

**Предусловие:** Система запущена, база данных пустая.

| № | Шаг | Ожидаемый результат | Фактический |
|---|-----|---------------------|-------------|
| 1 | Открыть http://localhost:5173 | Редирект на /login | |
| 2 | Нажать «Зарегистрироваться» | Форма с полями: имя, email, пароль | |
| 3 | Ввести корректные данные, нажать «Зарегистрироваться» | Сообщение «Регистрация успешна», редирект на /login | |
| 4 | Войти с теми же данными | Редирект на /dashboard, видно имя пользователя в sidebar | |
| 5 | Обновить страницу (F5) | Остаёмся на /dashboard (сессия сохранена) | |
| 6 | Нажать «Выйти» | Редирект на /login, cookie удалена | |
| 7 | Попробовать вручную перейти на /dashboard | Редирект на /login | |

**Критерии приёмки:**
- [ ] Регистрация с существующим email → сообщение об ошибке, не 500
- [ ] Пароль короче 6 символов → ошибка валидации
- [ ] Сессия сохраняется между обновлениями страницы
- [ ] Выход уничтожает сессию

## АС-002: Загрузка документа и AI-анализ

**Цель:** Пользователь может загрузить PDF/DOCX и получить AI-анализ.

**Предусловие:** Пользователь авторизован. Ollama запущена с моделью.

| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 1 | Перейти на «Загрузить документ» | Drop-zone с подсказкой |
| 2 | Загрузить PDF (drag & drop или кнопка) | Preview файла: имя и размер |
| 3 | Попробовать загрузить .exe | Ошибка «Только PDF и DOCX» |
| 4 | Попробовать загрузить файл > 15 МБ | Ошибка «Файл слишком большой» |
| 5 | Нажать «Отправить на анализ» | Редирект на «Мои заявки», статус «⏳ Анализ AI» |
| 6 | Дождаться завершения (обновить страницу) | Статус «✅ AI готов» |
| 7 | Открыть заявку | Отчёт с разделами: краткое описание, риски, условия, простым языком |

**Критерии приёмки:**
- [ ] Поддерживаются PDF и DOCX
- [ ] Файлы неверного формата отклоняются до отправки
- [ ] Статус заявки обновляется после анализа
- [ ] Отчёт содержит минимум: summary, хотя бы 1 раздел

## АС-003: Отправка заявки юристу

**Цель:** Пользователь может запросить дополнительную проверку юристом.

**Предусловие:** Есть заявка со статусом «AI готов».

| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 1 | Открыть заявку со статусом «AI готов» | Видна кнопка «Отправить юристу» |
| 2 | Нажать кнопку | Статус меняется на «Ожидает юриста» |
| 3 | Обновить страницу | Кнопка исчезла, статус сохранён |
| 4 | Попробовать повторно нажать (через URL) | Ошибка «Уже отправлено» |

**Критерии приёмки:**
- [ ] Кнопка отображается только при статусе ai_done
- [ ] Двойная отправка невозможна
- [ ] Заявка появляется в панели юриста

## АС-004: Работа юриста — взятие и проверка заявки

**Цель:** Юрист может взять заявку, отредактировать отчёт и сохранить заключение.

**Предусловие:** Юрист авторизован. Есть заявка в статусе «Ожидает юриста».

| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 1 | Войти как юрист | Sidebar с разделами: «Новые заявки», «В работе», «Завершённые» |
| 2 | Перейти «Новые заявки» | Видна заявка пользователя |
| 3 | Открыть заявку | Форма с AI-отчётом, редактируемые поля |
| 4 | Изменить уровень риска с medium на high | Поле изменилось |
| 5 | Добавить комментарий юриста | Текст введён |
| 6 | Нажать «Сохранить заключение» | Toast «Заключение сохранено», заявка в разделе «Завершённые» |
| 7 | У пользователя открыть ту же заявку | Виден раздел «Заключение юриста» с изменёнными данными |

**Критерии приёмки:**
- [ ] Юрист видит только заявки отправленные юристам (не pending_ai)
- [ ] Данные AI-отчёта подставляются как черновик в форму юриста
- [ ] После сохранения заявка недоступна в «Новых» — только в «Завершённых»
- [ ] Пользователь видит итоговый отчёт юриста

## АС-005: Разграничение доступа по ролям

**Цель:** Пользователи не имеют доступа к чужим данным и чужим функциям.

**Предусловие:** Созданы пользователи с ролями: user, lawyer, admin.

| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 1 | Войти как user, перейти на /admin | Редирект, доступ запрещён (403) |
| 2 | Войти как user, перейти на /lawyer | Редирект, доступ запрещён |
| 3 | Войти как user A, взять UUID заявки user B, открыть её | 404 или 403 |
| 4 | Войти как lawyer, перейти на /admin | 403 |
| 5 | Войти как admin, открыть любую заявку | Доступ есть |

**Критерии приёмки:**
- [ ] Все 5 сценариев выше возвращают ожидаемый результат
- [ ] Нет возможности угадать UUID чужой заявки (формат UUID4)

## АС-006: Панель администратора

**Цель:** Администратор управляет пользователями системы.

**Предусловие:** Авторизован admin@lexanalytica.ru.

| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 1 | Перейти в «Статистика» | 4 карточки: пользователи, заявки, ожидают юриста, проверено |
| 2 | Перейти «Пользователи» | Таблица со всеми пользователями |
| 3 | Изменить роль user → lawyer в dropdown | Toast «Роль изменена» |
| 4 | Нажать «Блок.» у пользователя | Статус меняется на «Заблокирован» |
| 5 | Попробовать заблокировать себя | Ошибка «Нельзя заблокировать себя» |
| 6 | Войти под заблокированным пользователем | Ошибка «Аккаунт заблокирован» |

**Критерии приёмки:**
- [ ] Смена роли работает и сохраняется после обновления страницы
- [ ] Заблокированный пользователь не может войти
- [ ] Admin не может заблокировать сам себя

## АС-007: Безопасность — SQL-инъекции

**Цель:** Система устойчива к попыткам SQL-инъекций.

| № | Payload | Эндпоинт | Ожидаемый результат |
|---|---------|----------|---------------------|
| 1 | `' OR '1'='1` | POST /api/auth/login (email) | 401 Unauthorized |
| 2 | `'; DROP TABLE users; --` | POST /api/auth/login (email) | 401, таблица users существует |
| 3 | `' UNION SELECT * FROM users --` | POST /api/auth/login | 401, данные не раскрыты |
| 4 | `<script>alert(1)</script>` | POST /api/auth/register (full_name) | 200/400, XSS не выполнен |

**Критерии приёмки:**
- [ ] Ни один payload не возвращает 200 с данными других пользователей
- [ ] Ни один payload не вызывает 500 Internal Server Error
- [ ] Таблица users существует после всех тестов

## АС-008: Производительность и доступность

**Цель:** Система обеспечивает приемлемое время отклика под нагрузкой.

**Инструмент:** locust -f tests.py --users=20 --spawn-rate=2 --run-time=60s

| Метрика | Целевое значение | Фактическое |
|---------|-----------------|-------------|
| Среднее время ответа GET /api/requests | < 200 мс | |
| P95 время ответа GET /api/requests | < 500 мс | |
| Процент ошибок при 20 RPS | < 1% | |
| Время ответа GET /api/health | < 50 мс | |
| Доступность за период теста | > 99% | |

**Критерии приёмки:**
- [ ] Все метрики соответствуют целевым значениям
- [ ] Нет утечек памяти за 60 секунд теста
- [ ] После завершения теста система отвечает корректно

## АС-009: Мониторинг и алерты

**Цель:** Система мониторинга фиксирует аварии и отправляет уведомления.

| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 1 | Открыть Grafana http://localhost:3000 | Все 4 дашборда загружаются |
| 2 | Проверить Executive дашборд | Статус «🟢 РАБОТАЕТ» |
| 3 | Остановить backend: `docker-compose stop backend` | — |
| 4 | Подождать 60 секунд | — |
| 5 | Проверить Prometheus /alerts | Алерт WebServiceDown = FIRING |
| 6 | Проверить почту администратора | Письмо «🚨 АВАРИЯ: Веб-сервис недоступен» |
| 7 | Запустить backend: `docker-compose start backend` | — |
| 8 | Подождать 60 секунд | Письмо «Resolved: WebServiceDown» |

**Критерии приёмки:**
- [ ] Алерт срабатывает в течение 90 секунд после отключения сервиса
- [ ] Email приходит на указанный адрес
- [ ] Письмо о восстановлении приходит после перезапуска

## АС-010: Логирование

**Цель:** Все ключевые действия фиксируются в логах, доступных через Grafana.

| № | Действие | Ожидаемый лог |
|---|---------|--------------|
| 1 | Успешный логин | `User logged in: user@example.com role=user` |
| 2 | Неудачная попытка входа | `Failed login attempt for: user@example.com` |
| 3 | Загрузка документа | `Document uploaded: file.pdf by user_id=5` |
| 4 | Завершение AI-анализа | `AI analysis done for request_id=12 in 45.3s` |
| 5 | Смена роли администратором | `User ... role changed: user → lawyer by admin_id=1` |

**Проверка в Grafana:**
- Перейти: Explore → Loki → `{service="backend"}`
- Убедиться что все события выше находятся в логах

**Критерии приёмки:**
- [ ] Все 5 событий фиксируются в логах
- [ ] Логи доступны в Grafana через Loki
- [ ] Логи содержат временну́ю метку и уровень (INFO/WARNING/ERROR)
- [ ] Пароли и токены **не** попадают в логи
